# 02-资料处理模块

## 定位

资料处理模块负责项目资料包的自动盘点和重点处理。它不写正文，不做影响判断。

资料处理模块只负责把资料变成后续模块可读、可定位、可复核的材料。它可以识别文件类型、页码、正文页、附件页、文本块、表格和图件，但不抽取“保护要求是什么”，也不判断项目是否符合保护要求。

默认策略：先自动盘点，再挑重点处理。

## 优先处理资料

必处理：

- 可研、项目建议书、工程建设方案、初设图纸等建设依据。
- 发改批复、立项选址、用地范围、宗地图、坐标表、KML、总平图、正射影像。
- 文物核查函、调查报告、勘探报告、验收意见。
- 文物本体资料、保护规划、保护范围、建控地带、缓冲区资料。
- 文物相关法规条例、管理办法、保护规划文本中关于保护范围、建控地带、缓冲区建设控制要求的条款。

可选增强资料：

- 既有文评。
- 既有保护方案。
- 评审意见。

没有可选增强资料不算缺项。

只登记不深处理：

- 商务资料。
- 合同。
- 询价函。
- 营业执照。
- 法人身份证。

默认跳过：

- `.DS_Store`。
- 明显重复件。
- 压缩包中的重复 KML，除非主文件打不开。

## 联网补充规则

项目包内法规、条例、保护规划和管控依据不足时，允许联网补充，但只用作外部依据。

联网搜索动作统一归资料处理模块负责。事实抽取模块只消费资料处理阶段已经保存的联网来源，不自行重新搜索；分析判断和报告拼装模块均不得临时联网补资料。

默认预置搜索清单：

1. 项目所在镇级行政区，或资料不足时上一级行政区的区域概况资料：
   - 地理位置。
   - 自然环境概况，包括地形地貌、气候、水文、生态等。
   - 社会经济情况。
   - 交通情况。
   - 历史沿革。
2. 本项目所涉及文物的相关法律法规、保护规划、管理办法、专项规定、主管部门文件或其他可作为文评依据的正式文件。

上述搜索宁可全面，不在资料处理阶段过早删减。资料处理模块应尽量多保存可靠来源和可用片段；后续事实抽取负责抽取结构化事实和条文，报告拼装阶段再按文章需要筛选、压缩和组织正文。

优先来源：

- 国家文物局、地方文旅厅、地方文物局官网。
- 政府公报、人大、司法行政部门法规库。
- 世界遗产中心或遗产地官方资料。
- 正式保护规划、管理条例、批复文件。

联网资料必须写入 `processing_output/external_sources.jsonl`，记录标题、发布单位、网址、发布日期或施行日期、访问日期和适用原因。

## 用户打扰规则

资料处理模块不得把“资料不完整”机械输出给用户。

只有在项目包、可替代资料、长期知识库和官方联网来源均无法补足，且会影响项目基本事实、空间边界、文物对象或核心管控依据时，才生成用户可见的不可替代资料缺项。

## 机器输出

```text
processing_output/
├── manifest.json
├── files.csv
├── processing_tasks.jsonl
├── text_index.jsonl
├── text_chunk_index.jsonl
├── table_index.jsonl
├── figure_index.jsonl
├── external_sources.jsonl
├── issues.jsonl
├── user_blocking_gaps.jsonl
├── extracted_text/
├── extracted_tables/
├── extracted_figures/
└── debug/
```

Markdown 只放在 `debug/`，不作为后续模块主输入。

正式文件、红头文件和主管部门函件应在资料处理阶段尽量切分正文页和附件页：

- `body_page_range`：正文首页至正文末页，第三章支持性文件插图优先使用该范围。
- `attachment_page_range`：附件、附表、附图等页，默认不进入第三章支持性文件正文插图，除非用户或项目规则明确要求。
- 正文页无法自动确认时，写入 `processing_output/issues.jsonl`，不得把单页占位图当作完整红头正文。

文号、发文单位、文件题名和日期可以在本阶段作为元信息初识别，但只作为证据台账登记线索；文号不准或不全时，不得静默补写进正文。

## 可用脚本

正式 v0.2.2 资料处理入口：

```bash
python scripts/run_material_processing.py --project-dir "<项目资料目录>" --output-dir "<输出目录>" --project-name "<项目名称>"
```

该脚本必须生成 `processing_output/`、`evidence/`、`module_state.json`、`run_state/material_processing.module_done.json` 和下一阶段提示。无不可替代缺项时生成 `next_prompts/next_prompt_fact_extraction.md`；有不可替代缺项时生成 `next_prompts/blocked_user_input.md` 并暂停。

辅助脚本：

```bash
python scripts/scan_project_folder.py --项目资料目录 "<项目资料目录>" --输出目录 "<输出目录>/debug/scan" --项目名称 "<项目名称>" --覆盖
python scripts/extract_source_text.py --项目资料目录 "<项目资料目录>" --输出文件 "<输出目录>/debug/priority_source_excerpt.md"
```

说明：

- `scan_project_folder.py` 当前生成 Markdown 底稿，可作为 `files.csv`、资料分类和章节映射的人工/AI 转换来源。
- `extract_source_text.py` 当前生成优先资料摘录，可作为后续事实抽取的片段来源。
- v0.2.2 正式机器产物必须落到 `processing_output/*.jsonl`、`*.csv`、`extracted_text/`、`extracted_tables/`、`extracted_figures/` 和 `evidence/`，不要只停留在 Markdown 底稿。

## 阶段硬隔离输出

资料处理阶段结束时必须写入：

```text
module_state.json
run_state/material_processing.module_done.json
next_prompts/next_prompt_fact_extraction.md
```

`next_prompt_fact_extraction.md` 必须说明：

- 项目工作包路径。
- `processing_output/manifest.json` 路径。
- `evidence/` 路径或待建立状态。
- 是否存在 `user_blocking_gaps.jsonl`。
- 下一阶段只能读取事实抽取模块规则、机器输出索引和必要原文片段。

若 `user_blocking_gaps.jsonl` 非空，不得生成事实抽取启动提示，应生成 `next_prompts/blocked_user_input.md`。

## OCR

需要 OCR 时，默认使用本地 Paddle OCR，路径为：

```text
/Users/drevan01/Desktop/OCR
```

PDF 文本处理必须先进入页级或片段级文本索引，不得只建立文件级证据。推荐顺序：

1. `pdfplumber` 直接抽取。
2. 本机 `pdftotext` 抽取。
3. `pypdf` 抽取。
4. 仍无文字时，尝试调用 `/Users/drevan01/Desktop/OCR` 的本地 Paddle OCR 服务。

若本地 Paddle OCR 服务未启动，应在 `processing_output/issues.jsonl` 记录内部 OCR 待处理问题，不默认打扰用户；但不得把该 PDF 当作已经完成文本处理。

## 全量抽取与切分

资料处理默认全量抽取已提供项目资料的文本内容，不设置默认字符截断上限。`--每文件最大字符` 只作为人工指定的应急参数；默认值 `0` 表示不截断。

材料内容较多时，不得先截断再抽事实。应完整写入 `processing_output/extracted_text/`，再按固定长度切分到 `processing_output/extracted_text_chunks/`，并在 `processing_output/text_chunk_index.jsonl` 记录每个切片的来源文件、起止字符和路径，供后续事实抽取或人工复核按片段读取。
