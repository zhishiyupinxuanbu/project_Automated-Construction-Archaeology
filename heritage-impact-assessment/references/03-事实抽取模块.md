# 03-事实抽取模块

## 定位

事实抽取模块从资料处理产物和证据台账中抽取可进入报告的事实材料。它不做专业影响判断。

## 输出

```text
facts/
├── project_facts.jsonl
├── heritage_facts.jsonl
├── requirement_facts.jsonl
├── quote_candidates.jsonl
├── source_coverage.jsonl
└── fact_issues.jsonl
```

## 阶段脚本

正式 v0.2.2 事实抽取入口：

```bash
python scripts/run_fact_extraction.py --workspace "<资料处理输出目录>"
```

脚本只能读取资料处理产物、证据台账和必要索引，输出 `facts/`、`run_state/fact_extraction.module_done.json` 和 `next_prompts/next_prompt_chapter5_pairing.md`。脚本产物不等于最终专业判断；AI 后续可按本模块规则和正式成稿专业 reference 对低置信度字段进行局部复核，但不得回到整包通读模式。

## 硬隔离输入

事实抽取阶段可在新上下文中启动。启动时只读取：

- `next_prompts/next_prompt_fact_extraction.md`
- `module_state.json`
- `processing_output/manifest.json`
- `processing_output/files.csv`
- `processing_output/text_index.jsonl`
- `processing_output/text_chunk_index.jsonl`
- `processing_output/table_index.jsonl`
- `processing_output/external_sources.jsonl`
- `evidence/evidence_register.jsonl` 或用于建立证据台账的索引
- 本文件

不得读取上一阶段聊天记录。不得跳过 `text_index.jsonl` 中已经提供的项目资料文本；所有非空间标记类文本至少必须进入 `source_coverage.jsonl`，记录是否抽取到结构化事实或可引用候选段。需要控制上下文时，可按 `text_chunk_index.jsonl` 逐片读取、逐份抽取，不得只挑一份“最像项目资料”的文本替代全量抽取。

## 必用专业 reference

正式文评事实抽取时，按需读取字段和资料识别规则：

```text
references/07-项目资料识别规则.md
references/08-事实抽取字段表.md
references/09-章节证据映射规则.md
references/11-固定正文结构与固定内容.md
```

这些 reference 用于保证抽取字段覆盖正式成稿需要，不得只抽“项目概况、文物概况、空间关系”几个摘要字段。

## 项目事实

包括但不限于：

- 项目名称。
- 建设单位。
- 建设地点。
- 建设内容。
- 用地范围。
- 建设规模。
- 施工内容。
- 建设必要性。
- 选址必要性。
- 建设目标。
- 投资规模。
- 资金来源。
- 建设工期。
- 建设模式。
- 主要经济技术指标。

事实抽取必须遍历所有已经提供并进入 `text_index.jsonl` 的项目文本来源，包括可研、项目建议书、工程建设方案、规划图说明、开工请示、调查勘探报告、验收意见和汇总材料。不得只从第一份或关键词匹配最高的一份资料中抽取项目事实。

文物调查报告、文物核查报告、考古调查勘探报告、文物主管部门函件属于项目事实来源，不属于“既有文评结论”禁用范围。其项目基础段若包含项目位置、建设内容、建筑面积、占地面积、拟用地范围与保护范围/建设控制地带/缓冲区关系，应同时抽入 `project_facts.jsonl` 和 `quote_candidates.jsonl`，优先供“一、总则 / （一）编制背景 / 1.评估项目基础信息”使用。

调查回函、文物调查报告、核查报告或主管部门函件中出现 `经查阅……发现……因此……文物影响评估` 这类总起段时，必须抽入 `quote_candidates.jsonl`，`quote_type=第二章开头总起段`，`target_section=二、建设项目涉及文物概况`。该候选用于第二章标题下的开头总起段，不得因后文已有“文物所在区域概况”而删除。

第三章和第四章的原文章节必须定向抽取：

- `（二）项目建设必要性`：从可研、可行性研究报告、建设方案或项目说明中定位“建设必要性”等对应章节，保留完整论证结构和分条理由。
- `（四）建设及运营方案`：从可研、可行性研究报告或建设方案中定位建设规模、技术指标、运营方案、运营规模、安全保障等章节；不得只抽项目概况中的零散字段替代。
- `（二）【项目名称】选址分析`：从可研、可行性研究报告或建设方案中定位选址分析、选址必要性分析、方案比选等对应章节，作为可引用叙述材料交给报告拼装。

## 文物事实

包括但不限于：

- 文物名称。
- 级别。
- 时代。
- 类型。
- 价值特征。
- 保存现状。
- 保护范围。
- 建控地带。
- 缓冲区。
- 管控要求。

第二章文物对象事实的抽取边界：

- `自然概况` 只抽地形地貌、气候、水文、生态等自然背景，不夹带过多文物内容。
- `遗址现状评估` 应抽具体保存现状事实，如墙体、遗存形态、病害、周边环境、已知扰动等，不抽空泛评价。
- `调查、发掘、保护工程情况` 只抽既有文物工作史，如三普、四普、历次调查、考古发掘、保护工程、公布为某级文保单位等；不得抽本项目文物调查、勘探工作替代本节。
- `价值陈述` 先抽项目资料中的价值线索；资料不足时标记需要按历史价值、文化价值、社会价值、军事价值、景观风貌价值等适用维度由报告拼装阶段成段撰写，不在事实抽取阶段硬编结论。

## 第五章配对页面驱动的条文抽取

项目事实、文物事实、空间关系事实和可引用叙述材料仍按本模块常规规则完整抽取；只有抽取法规、规划、保护要求、专项管控条文时，才按已抽项目事实、文物类型和实际涉及区划定向抽取。事实抽取模块只输出项目事实和条文两组分离数据，不输出合规结论或影响程度。

执行顺序：

1. 先完成项目事实、文物事实和空间事实抽取。
2. 按固定兜底清单补查可用于第五章判读的工程事实：地理位置、建设面积、建筑高度、建筑颜色、建筑形制、施工方式、施工期污染、施工期震动、施工期下挖、运营模式、运营期污染。
3. 判断哪些已抽项目事实需要对应条文支撑，例如建筑高度、建筑密度、用地类型、景观风貌、污染排放、施工扰动、地下文物风险等。
4. 只针对这些问题，从用户指定或资料中明确的法规、规划、保护规划、管理办法、主管部门文件中抽取具体条文、条款或控制要求。
5. 输出分离的 `project_facts.jsonl` 和 `requirement_facts.jsonl`，供 `scripts/generate_chapter5_pairing_page.py` 生成 HTML 匹配页面；不得在本阶段写“因此不符合规定”等结论。

元上都遗址专项规则：凡项目资料、文物事实或空间关系事实中出现元上都遗址、上都遗址、元上都遗址遗产区、元上都遗址缓冲区、正蓝旗上都工业园区、多伦县砧子山古墓群等线索，必须读取 `assets/17-元上都遗址保护管理规划关键条文.md`。该资产固定收录《元上都遗址保护管理规划修编·规划文本》第33条中规划文本第65-70页的连续管理规定，并已按 `zone:*` 和 `topic:*` 打 tag。事实抽取时应采用“三层基础包 + 主题补充”：具体类别条文 + 遗产区/缓冲区通用条文 + 元上都全域通用条文；例如一类缓冲区应调取 `zone:buffer_class_1`、`zone:buffer` 和 `zone:all`，再按项目事实补充 `topic:*` 条文。不得因涉及元上都而全量展开该资产。若只识别到“遗产区”或“缓冲区”但不能细分类型，只抽取对应通用 tag，并在 `fact_issues.jsonl` 记录需确认区划分类。

`requirement_facts.jsonl` 建议字段：

```text
requirement_id
requirement_source_title
requirement_source_evidence_id
requirement_location
requirement_text
control_object
control_value
applies_to
related_project_fact_ids
notes
```

示例输出边界：

```text
项目事实：鼓楼总高12米。
对应条文：《长城保护规划》第X章第X条规定，建设控制地带内建筑高度不得超过9米。
待分析问题：建筑高度与建筑密度是否符合保护规划要求。
```

不得在事实抽取阶段写“因此不符合规定”；该结论只属于分析判断模块。

## 可引用叙述材料

以下内容不按普通短字段处理，应进入 `quote_candidates.jsonl`：

- 项目背景。
- 建设必要性。
- 选址必要性。
- 文物价值描述。
- 保存现状。
- 保护要求。
- 法规管控要求。
- 调查类材料中可直接支撑第一章“评估项目基础信息”的项目位置、建设内容、面积、涉及保护区划等完整事实段。
- 项目可研报告、建设方案、设计文件或施工方案中 `4.技术规范` 对应章节的完整原文；该类候选写入 `quote_type=技术规范原文摘录`，供报告拼装阶段原文摘录，不得改写、概述或选择性摘抄。
- 项目可研、建设方案或项目说明中的建设必要性、建设及运营方案、选址分析对应章节；该类候选应保留章节结构和关键分条，不得压缩成一句概述。
- 红头文件、主管部门函件或批复的正文页完整文本；附件页可作为附件候选，不替代正文页。

可引用叙述材料的使用方式：

```text
可直接照搬
需改写
只可参考
需内部复核
不可引用
```

KML、OVKML、XML、坐标表、红线文件等空间结构化材料不得进入 `quote_candidates.jsonl`，即使其中包含项目名、点位名或坐标说明。它们只能用于空间关系字段、坐标/边界索引、图件索引或待量算事项。

事实抽取必须按 `text_index.jsonl` 的 `file_id` 回查 `evidence_register.jsonl` 中同一 `source_file_id` 的证据编号，不得默认把所有事实或引用材料挂到第一条证据 `E0001`。若没有匹配证据编号，该事实只能标为 `需内部复核`，并写入 `fact_issues.jsonl`，不得进入正式正文。

## 资料覆盖台账

`source_coverage.jsonl` 用于防止“文件已提供但事实抽取没有读进去”。每条非空间标记类文本来源至少一条覆盖记录。

字段：

```text
source_file
text_id
text_path
char_count
fact_count
quote_candidate_count
status
notes
```

`status` 取值：

```text
covered
no_structured_fact
```

若 `status=no_structured_fact`，必须同步写入 `fact_issues.jsonl`，说明该资料已读取但未抽取到结构化事实或可引用候选段，供下一阶段或人工复核。

## 既有报告边界

已有文评、已有保护方案、园区评估里的内容，默认只抽“事实”和“表达参考”，不抽“结论”。

该边界不得扩大到文物调查报告、核查报告、考古调查勘探报告或主管部门函件。上述调查审查类材料是本项目事实来源，不能因为禁止套用既有文评/保护方案结论而被排除。

例如正蓝旗中央厨房项目：

- 园区评估可提供背景、文物资料、保护范围或缓冲区依据、园区与遗产地关系、表达参考。
- 园区评估不得直接提供厨房项目本身的影响结论、可行性结论、保护措施结论。
- 判断性内容只能标注为 `只可参考` 或 `待分析`。

## 冲突处理

采用“来源优先级 + 冲突保留”。

默认优先级：

1. 正式批复、主管部门函件、核查意见。
2. 坐标表、KML、宗地图、规划图纸等空间边界文件。
3. 文物保护规划、管理条例、官方公布资料。
4. 可研、项目建议书、工程建设方案、初设图纸。
5. 调查报告、勘探报告。
6. 既有文评、保护方案。
7. 合同、询价函、商务资料只作辅助。

较新的正式文件可优先于旧文件；非正式过程稿不得自动覆盖正式批复。

冲突记录字段：

```text
issue_id
field_name
preferred_value
conflicting_values
preferred_reason
impact_level
requires_user_confirmation
```

## 阶段硬隔离输出

事实抽取阶段结束时必须写入：

```text
run_state/fact_extraction.module_done.json
next_prompts/next_prompt_analysis.md
```

`next_prompt_analysis.md` 必须说明：

- `facts/` 路径。
- `evidence/` 路径。
- `processing_output/external_sources.jsonl` 路径。
- 事实冲突与内部复核事项文件路径。
- 下一阶段只能读取分析判断模块规则、事实产物、证据台账、官方依据和必要方法论。
