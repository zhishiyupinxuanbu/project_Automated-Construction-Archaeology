# 19-第五章事实条文匹配页面与影响因子规则

## 定位

v1.0 将第五章输入闸门从“人工预填 Excel 表”替换为“事实与条文人工匹配 HTML 页面”。

用户给项目资料并要求生成文物影响评估报告时，skill 默认先完成资料处理和事实抽取。事实抽取阶段分别输出项目事实与规章条文，不输出合规结论、影响程度或专业判断。随后调用 `scripts/generate_chapter5_pairing_page.py --工作目录 "<工作包>"` 生成 `human_input/chapter5_fact_rule_pairing.html`，暂停等待用户完成人工配对。

旧版 `第五章人工预填表.xlsx` 只作历史兼容资源，不得作为默认流程入口。

## HTML 匹配页面

页面默认读取：

```text
facts/project_facts.jsonl
facts/requirement_facts.jsonl
```

默认输出：

```text
human_input/chapter5_fact_rule_pairing.html
human_input/chapter5_fact_rule_pairing_seed.json
human_input/chapter5_fact_rule_pairing_seed.csv
```

用户在页面中导出的人工结果应保存或整理为：

```text
human_input/chapter5_fact_rule_pairs.json
human_input/chapter5_fact_rule_pairs.csv
human_input/chapter5_pairing_notes.md
```

页面至少包含：

- 项目事实列表：事实编号、字段/主题、事实内容、证据编号、来源文件和位置。
- 规章条文列表：条文编号、控制对象、条文原文、控制值、证据编号、来源文件和位置。
- 配对记录：事实编号、条文编号、对应正文表、固定影响内容、人工判读、影响程度和备注。
- 导出 JSON 和 CSV 的按钮。

## 人工配对边界

- AI 只做资料消化、事实摘录、条文摘录、编号和来源标注。
- 用户做人机配对：把事实编号与条文编号连起来，并可补充判读、影响程度、计算口径或边界备注。
- 用户尚未配对时，AI 不得独立把工程事实和条文配成风险问题、合规结论或影响等级。
- 没有规则锚点的工程事实只能留在项目事实摘录中，不得被提升为风险问题或判读结论。
- 条文侧不做大而全摘录，只按项目实际涉及的文物类型和空间区划筛选。未涉及区划的条文不展开。

## 项目事实兜底清单

事实抽取阶段应尽量覆盖以下字段；资料中确无对应内容时写入 `facts/fact_issues.jsonl`，不得编造：

```text
地理位置
建设面积
建筑高度
建筑颜色
建筑形制
施工方式
施工期污染
施工期震动
施工期下挖
运营模式
运营期污染
```

常规项目事实、文物事实、空间关系事实和可引用叙述材料仍按事实抽取模块完整抽取；本清单只用于保证第五章人工配对页面具备最低可判读事实。

## 条文抽取规则

事实抽取模块应先完成项目事实、文物事实和空间事实抽取，再按下列方法抽条文：

1. 识别项目实际涉及的文物类型、保护范围、建设控制地带、缓冲区、遗产区或专项管控区。
2. 识别需要规则锚点的项目事实，例如建筑高度、建筑密度、用地类型、景观风貌、污染排放、施工扰动、地下文物风险等。
3. 只针对这些事实，从法规、保护规划、管理办法、主管部门文件和官方依据中抽取具体条文、条款或控制要求。
4. 输出项目事实和规章条文两组分离数据，不输出“符合/不符合”“影响轻微”“风险较低”等判断。

`facts/requirement_facts.jsonl` 建议字段：

```text
requirement_id
requirement_source_title
requirement_source_evidence_id
requirement_location
requirement_text
control_object
control_value
applies_to
related_project_fact_ids
notes
```

示例边界：

```text
项目事实：鼓楼总高12米。
规章条文：《长城保护规划》第X章第X条规定，建设控制地带内建筑高度不得超过9米。
人工配对：F0008 + R0003，影响内容为“建筑高度与建筑密度”。
分析判断：12米 > 9米，因此该项不符合规定。
```

不得在事实抽取阶段直接写“因此不符合规定”；该结论只属于人工配对后的分析判断模块。

## 项目设计合规性评估映射规则

固定表名：`项目设计合规性评估`。表号不固定，由最终报告顺序生成。

固定列：

```text
建设阶段
影响内容
详细信息
《文物保护法》及《长城保护规划》规定
是否合法合规
```

固定行：

```text
方案阶段 | 项目流程
方案阶段 | 环境保护
方案阶段 | 景观保护
方案阶段 | 用地类型
方案阶段 | 建筑高度与建筑密度
```

映射：

- 配对事实内容 -> `详细信息`
- 配对条文内容 -> `《文物保护法》及《长城保护规划》规定`
- 人工判读或分析模块比对结果 -> `是否合法合规`

如果配对条文来源为《长城保护规划》或其他保护规划，表格依据列应写具体条款、章节或控制要求，不能只写文件名。

`是否合法合规` 只能填：

```text
符合
不符合
```

如果证据不足以判断，不得在正式表中写“待确认”；应列为内部阻断或 `analysis/chapter5_pairing_issues.jsonl`，待补充后再成表。

## 建设期和运营期影响评估映射规则

固定表名：`项目建设期影响评估`。

固定表名：`项目运营期影响评估`。

表号不固定，由最终报告顺序生成。

固定列：

```text
建设阶段
影响内容
详细信息
对遗址本体的影响及程度
对环境风貌的影响及程度
对地下文物的影响及程度
```

`项目建设期影响评估` 固定行：

```text
施工阶段 | 施工过程对环境的影响（包括污染排放、地质灾害、水土流失、生态环境）
施工阶段 | 施工过程对地下不明文物的扰动
```

`项目运营期影响评估` 固定行：

```text
运营阶段 | 污染排放与生态环境影响
运营阶段 | 整体建筑风貌与视觉景观影响
```

映射：

- 配对事实内容 -> `详细信息`
- 人工影响程度或分析模块初步判断 -> 三类影响程度字段。
- 备注中的计算口径、控制条件和边界提示 -> 自然段分析要点或内部复核点。

## 固定影响因子

`项目建设期影响评估`、`项目运营期影响评估` 的三类影响程度只能从下列固定值中选择：

```text
有较大益处
有较小益处
正面影响可忽略
没有改变
负面影响可忽略
有较小负面影响
有较大负面影响
```

不得写：

```text
影响轻微
影响较小
基本无影响
可控影响
较低风险
待评估
待确认
不涉及
```

一般项目的 AI 初步判断最多到 `有较小益处` 或 `有较小负面影响`。只有人工明确说明，或项目事实和管控依据显示存在直接、显著影响时，才允许使用 `有较大益处` 或 `有较大负面影响`。

## 输出要求

分析判断阶段应将配对结果解析为机器文件，建议输出：

```text
analysis/chapter5_fact_rule_pairs.jsonl
analysis/chapter5_design_compliance.jsonl
analysis/chapter5_construction_impacts.jsonl
analysis/chapter5_operation_impacts.jsonl
analysis/chapter5_pairing_issues.jsonl
```

报告拼装阶段只能根据上述机器文件和 `assets/13-第五章影响评估表模板.md` 写入三张固定表，不得重新发明表结构。

第五章评估表格不能替代正文。分析判断阶段应把每项影响的自然段分析要点一并输出，报告拼装阶段先写正文分析，再放固定评估表。
