# 四有档案 RAG 工作流 v1.0

## 目标

建立“原始材料 -> 页级证据 -> 字段抽取 -> 候选卷别 -> 人工复核 -> 导出”的闭环。RAG 不是最终归档裁判，而是证据检索、相似材料召回、字段清洗和人工复核底座。

## 推荐工作区

在项目根目录内建立：

```text
00_RAG工作区/
  00_manifest/
  01_pages/
  02_ocr/
  03_text_clean/
  04_extract/
  05_classify/
  06_vector_db/
  07_review_html/
  08_exports/
  09_logs/
```

原始材料保留在原位置，不移动、不改名。工作区只保存处理产物、索引、证据、日志和导出。

## 处理顺序

1. 建总清单：遍历项目目录，排除 `00_RAG工作区`，登记所有原始文件。
2. 计算哈希：用文件哈希识别重复文件和重复图片。
3. 格式分流：
   - Word/WPS：优先抽文本；必要时转 PDF/页图再 OCR。
   - PDF：先尝试文本抽取；扫描件转页图 OCR。
   - 图片：保留原图、生成缩略图；含文字则 OCR；非文字图做标签/说明。
   - Excel/ET：保留表名、sheet、行列结构；关键表格抽成结构化数据。
   - DWG/BAK：登记为图纸候选，不直接归卷；需要人工确认或转图。
   - ZIP/RAR：先登记包内清单，展开后每个文件再入 manifest。
4. 页级证据：每页保留页图路径、OCR 文本、置信度、证据片段。
5. 字段抽取：按文件或独立文书抽题名、文号、日期、单位、文保点、乡镇、文件类型等。
6. 候选分类：按规则、字段、关键词、路径、向量召回、OCR 质量打分。
7. 人工复核：生成可打开原图/页图/OCR 原文的 HTML 或表格。
8. 导出：只导出人工确认后的结果；保留机器建议和人工修改痕迹。

## 日志要求

每次运行生成 `run_id`，写入 `09_logs/run-<timestamp>.json` 或 `.md`，至少包含：

- 运行时间、操作者/工具、脚本版本。
- 输入文件数、处理文件数、OCR 页数、失败项。
- 新增/更新的表。
- 人工确认数量和待核数量。
- 重要参数：权重配置、OCR 路径、向量模型或检索方式。
