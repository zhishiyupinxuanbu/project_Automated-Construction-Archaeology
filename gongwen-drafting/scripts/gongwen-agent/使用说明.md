# 公文撰写 Agent 使用说明

本文档用于说明 `/Users/drevan01/Desktop/公文撰写资料库` 中本地公文撰写 Agent 的运行环境、所需工具、输入内容、常用命令和输出结果。

## 一、工具定位

本 Agent 是资料库内的本地辅助工具，入口文件为：

```bash
agent/gongwen_agent.py
```

主要用途：

- 扫描资料库中的 `.docx` 和 `.pdf` 文件，建立本地索引。
- 从模板和历史参考件中检索同类材料。
- 根据项目要素、文种手册、地区模板和资料库固定版式规则，直接在桌面输出一份可复核使用的 Word 成稿。
- 对项目名称、建设单位、建设地点、项目面积、发文主体、主送机关等关键信息做基础一致性检查。
- 提供命令行和本地 Web 表单两种使用方式。

该工具不需要联网，生成内容主要依赖本资料库、用户输入字段和代码内置规则。正式报出前仍需人工复核事实、附件、受文单位和格式。

## 二、运行环境与所需程序

### 1. 基础环境

推荐在资料库根目录运行：

```bash
cd /Users/drevan01/Desktop/公文撰写资料库
```

需要：

- macOS。
- Python 3。
- `python-docx`：用于生成最终 `.docx`。
- Poppler 工具：
  - `pdftotext`：提取可复制文字 PDF。
  - `pdftoppm`：将扫描 PDF 转为图片，供 OCR 使用。
- OCR 工具：
  - 优先使用 `tesseract`。
  - 建议安装中文语言包 `chi_sim`、`chi_tra`，同时保留 `eng`。
  - 如无 `tesseract`，脚本会尝试使用 macOS Vision 备用 OCR；该方式需要 `clang` 编译 `agent/vision_ocr.m`。

当前机器已验证可用：

- `python3`：Python 3.9.6。
- `python-docx`：可导入。
- `pdftotext`：`/opt/homebrew/bin/pdftotext`。
- `pdftoppm`：`/opt/homebrew/bin/pdftoppm`。
- `tesseract`：`/opt/homebrew/bin/tesseract`，已包含 `chi_sim`、`chi_tra`、`eng`。
- `clang`：`/usr/bin/clang`。

### 2. 依赖检查命令

```bash
python3 --version
python3 -c "import docx; print('python-docx ok')"
which pdftotext
which pdftoppm
which tesseract
tesseract --list-langs
which clang
```

如缺少依赖，可根据本机环境安装。常见方式为：

```bash
brew install poppler tesseract tesseract-lang
python3 -m pip install python-docx
```

## 三、资料库文件要求

Agent 默认扫描整个资料库根目录下的 `.docx` 和 `.pdf` 文件，但会跳过 `.gongwen_agent` 缓存目录。

建议资料继续按现有结构放置：

- `1.政策法规与规范/`
- `2.公文模板/`
- `3.公文参考/`
- `agent/`

支持格式：

- Word：`.docx`
- PDF：`.pdf`

不支持或不建议：

- `.doc` 旧版 Word 文件。
- 图片散页直接入库。
- 文件名、正文高度混乱且无法识别文种的材料。

模板类文件名中如包含“模板”“模版”“XX”“XXX”等字样，Agent 会优先识别为模板。

## 四、首次建库与重建索引

索引会写入：

```text
.gongwen_agent/index.json
.gongwen_agent/texts/
```

### 1. 普通建库

适合资料多为可复制文字 PDF 和 Word 文件时使用：

```bash
python3 agent/gongwen_agent.py index
```

### 2. 推荐轻量 OCR 建库

适合先快速跑通工具，并让每类公文都有一定覆盖：

```bash
python3 agent/gongwen_agent.py index --ocr --ocr-max-pages 3 --coverage-per-type 5
```

含义：

- `--ocr`：对扫描 PDF 启用 OCR。
- `--ocr-max-pages 3`：每个 PDF 最多识别前 3 页，控制耗时。
- `--coverage-per-type 5`：每类公文最多入库 5 个文件，优先模板和已解析文件。

### 3. 强制全量 OCR 重建

资料更新较多、OCR 结果需要刷新时使用：

```bash
python3 agent/gongwen_agent.py index --force --ocr
```

注意：全量 OCR 可能耗时较长。扫描件较多时，优先使用轻量 OCR 命令。

## 五、检索模板和参考件

命令格式：

```bash
python3 agent/gongwen_agent.py search "项目或关键词" --type "事项类型" --region "旗区" -k 5
```

示例：

```bash
python3 agent/gongwen_agent.py search "纳林希里煤矿" --type "文物保护许可申请" --region "伊金霍洛旗" -k 5
```

可用事项类型通常包括：

- 文物保护许可申请
- 勘探验收请示
- 发掘请示
- 开工请示
- 勘探计划备案请示
- 勘探报告备案请示
- 文物保护安全责任书
- 文物调查意见
- 文物调查报告
- 其他

`--type` 和 `--region` 可不填，但填写后检索更准。

## 六、Web 表单使用

启动本地 Web 表单：

```bash
python3 agent/gongwen_agent.py web --port 8765
```

浏览器打开：

```text
http://127.0.0.1:8765
```

表单字段包括：

- 事项类型
- 发文主体
- 主送机关
- 项目名称
- 建设单位
- 建设地点
- 建设规模及主要内容
- 用地面积
- 审批/核准依据
- 联系人
- 电话
- 特殊说明

提交后会生成按资料库当前规则排版的 Word，并在页面上给出下载链接、事实校验结果和参考件列表。

## 七、命令行生成 Word 成稿

### 1. 准备项目 JSON

可参考 `agent/sample_project.json`。建议新建一个项目输入文件，例如：

```json
{
  "business_type": "文物保护许可申请",
  "issuing_org": "内蒙古示例能源有限公司",
  "recipient_org": "准格尔旗文物局",
  "project_name": "示例新能源配套110千伏输电线路工程",
  "construction_unit": "内蒙古示例能源有限公司",
  "location": "鄂尔多斯市准格尔旗大路镇境内",
  "scale": "新建110千伏输电线路约10公里，配套建设杆塔、光缆及相关附属设施",
  "land_area": "0.1200公顷",
  "approval_basis": "鄂尔多斯市能源局项目核准文件",
  "contact": "张三",
  "phone": "13800000000",
  "special_notes": ""
}
```

### 2. 既有公文格式调整

给已经写好的 `.docx` 公文调格式时，不再使用外部一键格式化工具覆盖全篇版式。应先判断文种、地区模板和当前文件问题，再按资料库内固定版式参数做外科式局部修正：

- 文物保护许可申请 / 核查请示：优先对照 `agent/固定版式说明.md`、`agent/make_hecha_docx.py` 和 `agent/check_hecha_docx.py`。
- 附件说明、落款、红头等固定区域：优先复用 `agent/gongwen_format.py` 中的段落缩进、制表位和落款对齐参数。
- 地区模板差异：先用 `agent/check_template_route.py` 确认是否存在地区专用模板，不得跳过本地模板改用通用口径。

### 3. 固定版式核查请示与专项生成

如需生成“文物保护许可申请/核查请示”，优先使用固定版式脚本：

```bash
python3 agent/make_hecha_docx.py agent/sample_hecha_project.json --out /tmp/核查请示.docx --audit
python3 agent/check_hecha_docx.py /tmp/核查请示.docx
```

固定版式脚本会把标题、正文、附件、落款分区生成，避免修改附件时误伤标题、修改正文缩进时破坏附件对齐。具体规则和字段见：

```text
agent/固定版式说明.md
```

项目中已保留固定版式样张：

```text
1.政策法规与规范/固定版式——文物保护许可核查请示格式范本.docx
```

红头、附件说明和落款是资料库专项生成工具中的固定版式区域。生成核查请示、按本资料库脚本大幅重建公文，或给既有成稿做局部格式修正时，均应优先复用 `agent/gongwen_format.py` 中的红头、附件和落款参数，不用外部一键格式化工具覆盖资料库既有版式。

### 4. 生成命令

```bash
python3 agent/gongwen_agent.py draft agent/sample_project.json
```

生成完成后，终端会输出 JSON 结果，包含：

- 识别出的事项类型。
- 选中的模板。
- 参考件列表。
- 最终格式化 Word 路径。
- 事实校验结果。
- 提醒信息。

## 八、输入内容填写要求

### 1. 最少建议填写

正式生成前，建议至少填写：

- `business_type`：目标文种或事项类型。
- `issuing_org`：发文主体。
- `recipient_org`：主送机关。
- `project_name`：项目名称。
- `construction_unit`：建设单位。
- `location`：建设地点。
- `scale`：建设规模及主要建设内容。
- `land_area`：项目面积或项目用地面积。
- `approval_basis`：备案、核准、批复、复函、报告等依据材料名称。

如缺少主送机关、项目面积、建设地点等关键信息，Agent 可能保留通用表述或占位，只能作为草稿，不宜直接作为终稿。

### 2. 字段说明

| 字段 | 含义 | 填写建议 |
| --- | --- | --- |
| `business_type` | 事项类型 | 尽量填标准类型，如“文物保护许可申请”“勘探验收请示”。 |
| `issuing_org` | 发文主体 | 企业、建设单位、项目单位或实际发文机关。 |
| `recipient_org` | 主送机关 | 不要提前写死，应按模板、复函、流程或用户要求确认。 |
| `project_name` | 项目名称 | 必须与材料原文保持一致。 |
| `construction_unit` | 建设单位 | 与发文主体可能相同，也可能不同。 |
| `location` | 建设地点 | 写到能判断旗区和项目所在地的程度。 |
| `scale` | 建设规模及主要建设内容 | 尽量按资料原文填写，不随意压缩关键建设内容。 |
| `land_area` | 项目面积或项目用地面积 | 使用“公顷”等原始或确认后的单位。 |
| `approval_basis` | 审批/核准/备案依据 | 可填备案告知书、核准文件、政府批复、复函、报告等名称。 |
| `contact` | 联系人 | 没有则留空，不写占位。 |
| `phone` | 电话 | 没有则留空，不写占位。 |
| `special_notes` | 特殊说明 | 写需要额外纳入正文或影响判断的情况。 |

### 3. 面积表述规则

输入和正文中统一使用：

- 项目面积
- 项目用地面积

禁止使用：

- 调查面积
- 勘探面积

如果资料中出现多种面积口径，应先确认，不要擅自合并或换算。原始单位为平方米时，可以折合为公顷，不折合为亩。

### 4. 勘探验收请示补充信息

生成“勘探验收请示”时，建议在 `special_notes` 或相关字段中补充：

- 项目实施目的或意义。
- 是否已委托第三方公司开展考古调查、勘探工作。正文不写具体勘探公司名称，即使资料中有具体单位，也统一写“第三方公司”或“第三方单位”。
- 是否未发现文化遗存、无遗迹。
- 如发现遗迹，报告中的直接结论，例如“共发现灰坑遗迹39处”。
- 报告名称或可确认的成果材料名称。

无遗迹情形不要主动写成果；有遗迹时只写报告直接结论，不展开逐项描述。

### 5. 附件信息

Agent 会优先从模板中提取附件提示；提取不到时使用内置默认附件。

正式使用前应核对：

- 附件是否确实提供。
- 附件名称是否与材料一致。
- 是否需要删去营业执照、宗地图、KML、法人身份证等未提供或不适用附件。
- 勘探验收默认只保留“考古调查、勘探工作报告”，不放勘探计划；除非用户明确要求或当地模板明确要求。

## 十一、生成后复核清单

输出前后建议重点检查：

- 项目名称是否前后一致。
- 发文主体、落款单位是否一致。
- 主送机关是否与项目所在地、办理阶段和材料依据匹配。
- 建设地点、项目面积、建设内容是否与原始资料一致。
- 是否误写“调查面积”“勘探面积”。
- 文种是否正确，是否把企业申请误写成文物部门内部转报。
- 附件名称和实际提供材料是否一致。
- 联系人、电话没有提供时是否已删除。
- 日期是否为当天或用户指定日期。
- Word 版式是否符合当前模板要求。

## 十二、常见问题

### 1. 生成失败：缺少 python-docx

说明 Word 生成依赖缺失。处理：

```bash
python3 -m pip install python-docx
```

### 2. PDF 解析结果少，提示 needs_ocr

说明 PDF 可能是扫描件或不可复制文字。处理：

```bash
python3 agent/gongwen_agent.py index --ocr --ocr-max-pages 3 --coverage-per-type 5
```

如仍不理想，再考虑：

```bash
python3 agent/gongwen_agent.py index --force --ocr
```

### 3. OCR 很慢

优先使用轻量模式：

```bash
python3 agent/gongwen_agent.py index --ocr --ocr-max-pages 3 --coverage-per-type 5
```

不要一开始就对大量扫描 PDF 做全量 OCR。

### 4. 参考件不准

优先补充：

- `business_type`
- `region`
- `project_name`
- `construction_unit`
- `location`

也可以先用 `search` 命令手动确认参考件。

### 5. 正文出现占位或通用表述

通常是输入字段缺失。补齐项目名称、建设单位、建设地点、项目面积、主送机关、发文主体后重新生成。

## 十三、边界说明

本 Agent 适合生成初稿和辅助检索，不替代人工审稿。涉及政策法规、审批要求、报送对象、文种边界、面积口径、附件清单和项目事实时，应以用户提供材料、正式模板和可核实来源为准。

对模板类 Word 文件进行正式修改时，应优先保持原模板结构和版式，只替换应替换内容，清除不应保留的高亮提示，不擅自增加无关内容。
