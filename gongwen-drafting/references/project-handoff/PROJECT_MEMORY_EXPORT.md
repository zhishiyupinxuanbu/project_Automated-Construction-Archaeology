# 项目记忆导出

本文件用于给迁移后的 Codex、Trae Solo 或其他 AI 编程助手快速接手。详细规则以 `AGENTS.md` 为准；不要再另建或依赖 `公文写作规则.md`，也不要依赖原电脑的 `~/.codex/memories`。

## 用户偏好

- 用户希望公文输出正式、稳健、可直接落文。
- 用户重视模板忠实度，尤其是模板类文件，通常只希望替换应替换内容，不希望擅自扩写。
- 用户纠正规则后，希望规则被固定下来，后续不再重复犯错。
- 对文物手续类文件，事实准确性优先于文采。
- 对缺项、矛盾、数据口径不一致的信息，应提醒确认，而不是自行推断。

## 资料库和 Agent 定位

- 资料库根目录是一个文档知识库，不是普通代码仓库。
- 自动化代码集中放在 `agent/` 目录。
- 主入口是 `agent/gongwen_agent.py`。
- 项目级长期记忆入口是 `AGENTS.md`。
- 面向人和其他 AI 助手的长期规则文件也是 `AGENTS.md`，保持一个入口，避免规则分叉。
- AI 专用公文撰写总流程 skill 在 `agent_handoff/skills/gongwen-drafting/SKILL.md`；迁移后生成、修改、路由或维护公文任务时先读该 skill，用于统一启动判断、文种路由、资料需求、Word 交付链路和资料库维护边界。
- AI 专用固定版式 skill 在 `agent_handoff/skills/gongwen-hecha-fixed-format/SKILL.md`；迁移后生成或修改核查请示时应同时读该 skill，用于按资料库固定版式直接生成 Word 成稿。
- `agent/format_existing_docx.py` 和 `agent/vendor/docformat_engine/` 已从当前交付链路移除，不再作为迁移包必要工具。
- Obsidian 索引在 `0.资料索引/`，Obsidian 配置在 `.obsidian/`。
- 模板在 `2.公文模板/`。
- 历史参考件在 `3.公文参考/`。
- 每日首次维护资料库或接手项目时，应检查 `3.公文参考/` 新增公文，按现有“文种—地区—文件角色”目录整理归档，并据此更新索引、文件卡片、文种手册、模板索引、模板内化、规则候选或健康检查；只有经多份参考件、明确地区模板或用户确认支撑的规律，才校正为通用/地区模板和版式硬规则。
- 政策规范在 `1.政策法规与规范/`。

## 跨设备迁移边界

- 迁移目标默认是另一台 Mac，接手 AI 是 Codex 或 Trae Solo。
- 本机 memories 中真正影响稳定输出的规则已经显化到 `AGENTS.md`、本文件、`agent_handoff/skills/` 和 `agent/` 脚本中；对外迁移时优先以 `gongwen-drafting` skill 包承载可迁移规则。
- 新电脑不需要复制原电脑的 `~/.codex/memories` 才能工作；如果保留 memories，只能作为只读追溯材料，不能作为执行主入口。
- 给人的说明保持短：导入或解压 `gongwen-drafting.skill`，让 AI 先读其中 `gongwen-drafting/SKILL.md`；如需继续维护原资料库，再另行复制完整资料库。
- 复杂业务规则不要写进人用 PPT 或口头说明，应维护在 `AGENTS.md`、本文件、skill 和脚本里。
- 后续迁移只保留“给 AI 助手迁移包”，且该迁移包本身就是一个 skill 包；不再制作“完整运行包”，也不再把完整资料库打入这个迁移包。默认迁移目标电脑已有 Codex、Trae Solo 等 AI 工具，无需打包 Windows 安装/启动文件。

## 迁移内容清单

给 AI 助手迁移包应包含：

- `gongwen-drafting/SKILL.md`
- `gongwen-drafting/references/`
- `gongwen-drafting/assets/`
- `gongwen-drafting/scripts/`
- `gongwen-drafting/agents/openai.yaml`

其中 `references/` 承载项目规则、知识索引、文种手册和模板内化，`assets/` 承载政策规范、Word 模板和历史参考件，`scripts/` 承载确定性工具链。

默认不迁移：

- `.gongwen_agent/`
- `__pycache__/`
- `.DS_Store`
- `.venv/`
- `config.local.json` 等本机配置
- 运行日志和 pid 文件
- 完整运行包、Windows 安装/启动包
- 未整理进 skill 内资源的完整资料库副本
- 临时输出、旧 zip 包、个人桌面成稿
- 只在某一次项目中使用的中间文件，除非它已经被整理为模板或案例

## 已验证的常用命令

```bash
python3 scripts/gongwen-agent/gongwen_agent.py index --ocr --ocr-max-pages 3 --coverage-per-type 5
```

```bash
python3 scripts/gongwen-agent/gongwen_agent.py web --host 127.0.0.1 --port 8765
```


## 格式化 Word 输出优先规则摘要

- 启动公文撰写流程后，只要能够基本判断文种、办理阶段和发文主体，应先基于现有材料生成一版可直接打开使用的格式化 Word。
- 正文不得留空，不得保留“XX”“20XX”“待补充”“请填写”等占位文字。
- 确无资料支撑的信息，应按文种规则省略、调整表述或使用现有材料中的稳妥概括，不在正文或附件栏留下空项。
- AI 按文种手册、地区模板、固定版式 Word 样张和内置脚本直接生成桌面 `.docx` 成稿；不得再调用 `docformat-gui` 或通用一键格式化工具覆盖既有版式。
- 最终调整版式时，必须以 `assets/fixed-format/固定版式——文物保护许可核查请示格式范本.docx` 为固定版式 Word 样张，逐项核对红头、标题、正文、附件、落款的字体字号、缩进、附件悬挂缩进、制表位和落款对齐。
- Markdown 草稿、未格式化 Word 和中间过程文件不作为交付物，除非用户明确索要。
- 格式化 Word 生成后，再集中询问用户是否有补充材料、是否需要调整受文单位、附件、面积口径、建设内容、联系人电话等。
- 只有在无法判断文种、办理阶段、发文主体或收文关系，且继续生成会导致行文方向明显错误时，才先向用户集中确认关键问题。

## 最新补充规则摘要

- 项目记忆边界：具体文件生成或修改中的一次性要求，不自动上升为长期规则；只有用户明确要求固定、记住、写入项目记忆或以后都按此处理时，才写入长期规则。
- 输入输出：AI 交付应以可直接打开使用的格式化 Word 文档为主；生成或修改后的 `.docx` 默认保存到电脑桌面，不放入资料库内部目录；缺项和风险在对话中说明，不写进正文占位。
- 勘探计划备案请示：呼伦贝尔市项目附件必须包含“考古调查、勘探工作计划”和勘探公司资质材料或备案资料；非呼伦贝尔市项目一般只放计划，不主动加资质、营业执照、法人身份证等。
- 申请开展考古勘探工作请示：用户说“勘探请示”“开展勘探请示”“申请勘探”等且未明确验收、成果备案、计划备案时，优先按“申请开展考古勘探工作请示”处理，不要误写成勘探验收请示。
- 考古勘探工作支持请示：用户说“协助勘探”“考古勘探工作支持”“上报考古院支持勘探”等，且项目涉及长城、金界壕等重点文物段落，或有资金压力、工期紧张、技术要求高、实施难度大等支持理由时，按独立文种处理，不并入普通“申请开展考古勘探工作请示”。默认模板为 `2.公文模板/模板——XX项目考古勘探工作支持的请示.docx`，默认由建设单位、项目单位或企业向属地文旅局/文物局请示，请其上报内蒙古自治区文物考古研究院或材料明确的专业机构提供支持；标题通常写“关于XX项目考古勘探工作支持的请示”，正文先写项目背景、批复或备案、项目代码、地点、建设内容、意义、文物复函和政策依据，再写困难及支持请求；面积写“涉及XX部分（约X平方米）”，不得写“调查面积”“勘探面积”，默认不写附件。
- 开工请示：属于文物手续流程最后一步，附件组件要求更强；默认应核对可行性研究报告、用地范围、经纬度坐标、宗地图、考古调查勘探报告、检查意见书等材料，缺项要提示用户补充或确认。伊金霍洛旗市级及以上项目使用“验收及开工请示”模板；旗级项目才使用只申请开工、不请示验收的开工请示口径；旧伊旗纯开工模板已从 skill 删除，不再调用。伊旗验收及开工请示附件按长庆油田参考件先列全，第一项为“立项核准文件”，后续为文物审查意见函、坐标、KML、宗地图、考古调查勘探报告、营业执照和负责人或法定代表人身份证复印件。
- 鄂尔多斯项目生成考古勘探意见书本体时，标题统一写“考古勘探项目检查意见书”；验收意见段不得写“XX公司对项目进行了考古勘探工作，”这类重复句。开工请示附件中即使实际文件名为“验收意见书”，也按“考古勘探项目检查意见书”口径书写。
- 文物调查报告、市级调查报告、自治区调查报告：已拆分到独立 `gongwen-survey-report` skill；本 `gongwen-drafting` skill 只负责请示、函、许可、备案、验收、开工等公文流程。
- Obsidian 索引：当前资料库已建立 `0.资料索引/`，含总览、按文种索引、模板索引、待补充文本清单和文件卡片；迁移包应包含 `.obsidian/` 和 `0.资料索引/`。
- 参考公文更新：每天维护或接手项目时检查 `3.公文参考/` 新增公文，归档后从文种、地区、发文主体、受文单位、正文结构、附件设置和版式特征中提炼规律；疑似个案先进入规则候选或健康检查，不直接覆盖硬规则。
- Skill 分层：复杂规则写进 `gongwen-drafting/SKILL.md`、`references/project-rules/AGENTS.md` 和 `scripts/gongwen-agent/` 脚本；人只需让 Codex 按 skill 规则接手。总流程 skill 负责路由和交付链路，专项生成工具负责 Word 的红头、标题、正文、附件和落款结构；最终成稿不得再由 `docformat-gui` 二次覆盖。

## 勘探验收请示稳定规则摘要

- 勘探验收保留新旧两种模板；默认使用新模板 `2.公文模板/模板——关于申请XX项目考古勘探验收的函.docx`，标题通常写为“关于申请XX项目考古勘探验收的函”。
- 只有用户明确要求旧模板、当地材料明确要求“文物勘查验收许可的请示”，或已有文件就是旧模板口径时，才写“关于XX项目用地选址范围内文物勘查验收许可的请示”。
- 不要加发文主体，不要改成“文物勘探成果验收”等自由标题。
- 正文通常 3 段左右，不逐条列项目基本信息。
- 第一段写项目实施目的或意义，并说明委托第三方开展文物工作。
- 第二段写项目位置、项目面积、建设内容。
- 第三段引用内文物发〔2025〕6号，说明完成实地考古勘查并形成报告。
- 正文不写具体勘探公司名称；即使报告中有具体单位，也统一写“第三方公司”或“第三方单位”。
- 使用“此函”收束时，“此函”必须单独成段，不为压缩页数并入上一段。
- 6号文只出现一次。
- 无遗迹时不写成果。
- 有遗迹时只写报告直接结论，不展开逐项描述。
- 附件默认只放勘探报告，不放勘探计划；报告附件必须写完整项目名称版全称，格式为“完整项目名称考古调查、勘探工作报告”或材料中的正式报告名称。附件栏按跨文种固定版式处理：第一项连续起排为“附件：1.”，附件名称续行与附件名称首字对齐。
- 没有联系人和电话时不写。

## 考古勘探工作支持请示稳定规则摘要

- 本类文件也可称为“协助勘探请示”“考古勘探工作支持请示”“恳请提供考古勘探支持的请示”。
- 触发场景是项目涉及长城、金界壕等重点文物段落，或材料显示资金压力、工期紧张、技术要求高、实施难度大，需要文物考古专业机构支持。
- 不并入普通“申请开展考古勘探工作请示”；普通请示重点是申请开展勘探，本类重点是请求属地文旅/文物部门上报专业机构提供支持。
- 默认模板为 `2.公文模板/模板——XX项目考古勘探工作支持的请示.docx`；写作前读 `0.资料索引/文种手册/考古勘探工作支持请示-作战手册.md` 和 `0.资料索引/模板内化/模板——XX项目考古勘探工作支持的请示.md`。
- 默认发文主体为建设单位、项目单位或企业；受文单位通常为属地文旅局、文化和旅游局或文物局，不直接把自治区文物考古研究院写成受文单位，除非材料或用户明确要求。
- 正文按“项目背景和批复手续—地点、建设内容、项目意义—文物复函和政策依据—涉及文物段落、困难和支持请求—收束语”组织。
- 请求句通常为恳请受文单位上报内蒙古自治区文物考古研究院或材料明确的专业机构，对项目建设区域涉及重点文物段落部分提供考古勘探支持，协助建设单位规范、高效完成相关范围内考古勘探工作。
- 面积写“项目建设区域涉及XX部分（约X平方米）”或“相关范围（约X平方米）”；不得写“调查面积”“勘探面积”，不得擅自折算为亩。
- 默认不写附件；没有联系人和电话时不写。

## 文物保护许可申请 / 核查请示稳定规则摘要

- 本类文件包括“文物核查请示”“文物手续”“文物保护许可”“许可申请”等。
- 默认只处理企业、建设单位、项目单位向文物部门申请或请示的文件。
- 不写成文物部门向上级主管部门报送的内部转报请示。
- 标题可按“XX公司关于办理XX项目用地范围内文物保护许可的请示”组织。
- 正文可保留模板条目结构。
- 锡林郭勒盟/西乌珠穆沁旗项目，用户简称“锡盟”“锡林郭勒盟”“西乌”“西乌珠穆沁旗”且办理文物保护许可、文物核查或申请办理文物调查事宜时，优先使用 `模板——锡林郭勒盟西乌珠穆沁旗文物调查事宜请示`，不回退通用模板。
- 锡林浩特市项目，用户简称“锡林浩特”“锡林浩特市”“锡市”，或项目名称/材料出现“胜利矿区”“胜利煤田”“胜利矿田”且指向锡林浩特市，办理文物保护许可、文物核查、查询用地是否涉及文物或临时用地涉文查询时，优先使用 `模板——锡林浩特市临时用地是否涉及文物请示`，不回退通用模板。该模板来源为纸质照片，遮挡处只作为可变项；受文单位通常为“锡林浩特市文体旅游广电局”，附件最小清单为“临时用地范围及电子版国家2000坐标”。如材料不是临时用地，保留地方模板口径但替换不符合事实的“临时用地”“临时车辆行驶道路”表述。
- 建设规模、建设内容尽量依据项目资料原文填写。
- 面积集中写在“项目面积”或“项目用地面积”处。
- 禁止写“调查面积”“勘探面积”。
- 一般不写建设期限，除非用户明确要求或材料显示必须说明。
- 核心申请句应为申请核查项目建设用地范围内是否涉及文物保护单位，并恳请出具相关审查意见。
- 附件按实际资料和用户要求增减，不机械套模板。

## 文物调查报告 skill 边界

- 用户说“调查报告”“文物调查报告”“文物调查的报告”“市级调查报告”“自治区调查报告”“联合调查报告”等，转用独立 `gongwen-survey-report` skill。
- 本 skill 不保留调查报告模板、调查报告案例、调查报告手册或调查报告模板内化文件。

## 常见风险

- 把企业申请误写成文物部门内部请示。
- 把“项目面积”误写成“调查面积”或“勘探面积”。
- 主送机关提前假定。
- 没有联系人电话却保留空占位。
- 模板类材料擅自扩写，破坏原模板结构。
- 把其他文种的措辞泄漏到当前文种中。
- 在本 skill 内继续处理文物调查报告，而不是转用 `gongwen-survey-report`。
- 对 PDF/OCR 解析结果过度信任，未回到用户原始材料核对。

## 接手原则

先读用户资料和模板，再写。能从资料中确认的直接使用；不能确认且影响正文事实、主送机关、附件一致性的，先问用户。输出时尽量给出可直接使用的版本。
