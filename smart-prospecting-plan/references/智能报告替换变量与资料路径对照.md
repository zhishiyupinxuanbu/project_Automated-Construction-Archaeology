# 智能报告替换变量与资料路径对照

本文档用于维护智能报告/计划模板中的有效替换变量、变量数据来源，以及对应的文件路径或文件名匹配规则。适用范围为当前智能模板生成线：

- 生成脚本：`智能生成报告技能资料/工具脚本/fill_smart_template_from_form.py`
- 表单预填脚本：`智能生成报告技能资料/工具脚本/create_manual_form_from_project.py`
- 基准模板目录：`智能生成报告技能资料/知识库/3.模板与表单/模板/报告、计划/`
- 表格模板目录：`智能生成报告技能资料/知识库/3.模板与表单/模板/表格模板/`
- 项目源资料目录：通常为 `基础信息/<项目目录>/`

本表只整理当前智能模板生成线，不整理旧模板生成线 `generate_report.py` 的 `项目信息.xlsx` 变量。`基础信息/` 为只读源资料区，不在其中写入本说明或生成成果。

## 路径占位说明

| 名称 | 含义 |
| --- | --- |
| `<项目名称>` | 人工填写表或项目源资料识别出的项目名称。 |
| `<项目目录>` | `基础信息/` 下的具体项目资料目录。 |
| `--form` | 已确认的人工填写表，通常为 `过程资料/<项目名称>/人工填写表/<项目名称>_人工填写表.xlsx`。 |
| `--drawings-dir` | 制图成果或内业成果目录，通常为 `基础信息/<项目目录>/3.执行资料/3.制图成果`，也可兼容 `3.执行资料/3.内业成果`。 |
| `--photos-dir` | 外业成果照片目录，通常为 `基础信息/<项目目录>/3.执行资料/1.外业成果`。 |
| `图片/` | 人工填写表同级目录下的图片包，即 `过程资料/<项目名称>/人工填写表/图片/`。 |

## 表单来源变量

普通文本变量优先从 `--form` 的键值工作表读取。人工填写表模板为：

`智能生成报告技能资料/知识库/3.模板与表单/模板/表格模板/人工填写表模板.xlsx`

项目专用智能报告信息填报表也以该人工填写表模板为底表生成，不再维护单独的 `智能报告信息填报表_空白模板.xlsx`。脚本会读取这些键值工作表：`项目基础信息`、`项目区域概况`、`文物概况`、`勘探参数`、`现场限制`、`人员构成`、`自动生成字段`。解析时允许用户把内容填在右侧补充列，不只限于 `值` 列。

## 表格工作表与重复块变量

脚本会读取这些表格型工作表：`红线坐标`、`勘探单元`、`剖线地层堆积`、`标准孔`、`遗迹记录`、`遗迹坐标`、`文物范围遗迹统计`、`图件照片目录`、`图片清单`。其中部分工作表可由 `--drawings-dir` 或 `--photos-dir` 自动导入。

专项检查中的 `表格替换检查` 会把这些本项目源表/导入表的期望记录数，与生成检查报告中记录的实际生成或填充计数进行比对，用于发现报告仍沿用模板旧表、旧资料表或只替换了部分记录的风险。期望记录数大于 0 但生成计数为 0、未记录或少于期望记录数时，应写入专项检查 XLSX 的 `错误与失败原因`，并在问题摘要中归入 `表格替换待复核`；剖线地层堆积按有效剖线编号去重后比较。

| 替换变量/重复块 | 数据来源 | 对应路径/名称规则 | 备注 |
| --- | --- | --- | --- |
| `{{#红线坐标}}...{{/红线坐标}}` | 人工填写表 `红线坐标`；为空时从制图表格导入 | 优先 `--drawings-dir/表格/红线四至坐标.xlsx`、`红线四至坐标表.xlsx`、`四至范围坐标.xlsx`、`四至范围坐标表.xlsx`；兼容这些文件在 `--drawings-dir` 根目录或递归子目录 | 生成四至范围坐标重复内容；兼容“标题行后直接数据”和“角点/X坐标/Y坐标表头 + 数据行”格式。 |

| `{{图:资料整理工作照}}`、`{{图:文献资料收集与整理工作照}}` | 外业照片 | `资料整理工作照` 匹配 `--photos-dir/**/资料整理工作照*` 或 `*资料整理照*`；`文献资料收集与整理工作照` 匹配 `*文献资料收集与整理工作照*` 或 `*文献资料收集*`，若无则复用 `资料整理工作照` | 文献资料收集照可后备复用资料整理照；模板中可能被 Word run 拆分，按可读变量维护。 |

| `{{自动图号}}` | 图片题注派生 | 题注段内使用，如 `图{{自动图号}} {{图:实地踏查照}}` | 插图后正文图号会重新连续编号。 |

## 图片清单覆盖规则

`图片清单` 工作表可人工覆盖图片匹配。常用列为：

| 列名 | 用途 |
| --- | --- |
| `图片位` 或 `插入位置` | 对应模板中的 `{{图:图片位}}`。 |
| `文件名` 或 `图片文件名` | 在 `--form` 同级 `图片/`、`--drawings-dir`、`--photos-dir` 中匹配图片。 |
| `图题` | 覆盖题注中显示的图片名称。 |

覆盖优先级高于自动关键词匹配。图片文件支持 `.jpg`、`.jpeg`、`.png`。

## 派生变量与计算规则

| 派生变量 | 输入来源 | 计算/匹配规则 | 备注 |
| --- | --- | --- | --- |
| `{{调查面积}}` | `项目面积` | 未填时默认等于项目面积 | 若二者不同，应人工填写。 |
| `{{项目所在地旗县}}` | `项目位置` | 从项目位置中识别 `旗`、`县`、`区`、`市` 结尾的旗县级名称 | 可被人工填写覆盖。 |
| `{{报告年月}}` | `结束日期`、`勘探时间` | 识别 `YYYY年M月` 或日期格式，输出 `YYYY年M月` | `结束日期` 优先。 |
| `{{勘探单元规格}}` | 人工填写值 | 纯数字 `100` 规范化为 `100米×100米` | 用于探孔编号尾号。 |
| `{{普探列最大编号}}`、`{{普探行最大编号}}` | `勘探单元规格`、`普探孔距` | `单元边长 / 孔距 * 2 - 1` | 例：400m、2m -> 399。 |
| `{{重点勘探列最大编号}}`、`{{重点勘探行最大编号}}` | `勘探单元规格`、`重点勘探孔距` | `单元边长 / 孔距 * 2 - 1` | 例：400m、1m -> 799。 |
| `{{勘探单元数量}}` | 勘探单元坐标记录 | 对 `勘探单元` 去重计数 | 自动导入外部表后会刷新。 |
| `{{勘探单元编号范围}}` | 勘探单元编号 | 同前缀连续且数量不少于 4 时，写为首两项加末项 | 例：`U01、U02……U131`。 |
| `{{技师数量}}` | `技师` 名单 | 按顿号、逗号、空格等分隔计数 | 人工填写优先。 |
| `{{探工数量}}` | `进场人数` | `进场人数 - 8` | 当前流程要求 `进场人数` 必填。 |
| `{{每日在场人员下限}}` | `进场人数` | `进场人数 - 5` | 小于 0 时按 0 处理。 |
| `{{踏查技师数量}}` | `技师数量` | 未填时取 `技师数量` | 用于踏查人员表述。 |
| `{{踏查探工数量}}` | `探工数量` | 未填时取 `探工数量` | 用于踏查人员表述。 |
| `{{剖线数量}}` | `剖线地层堆积` 表 | 记录数 | 未填时自动写入。 |
| `{{标准孔数量}}` | `标准孔` 表 | 记录数 | 以 `标准孔坐标.xlsx` 为主键表。 |
| `{{遗迹数量}}` | `遗迹记录` 表 | 记录数 | 有记录时可生成 `发现遗迹N处`。 |
| `{{勘探成果综合结论}}` | 剖线地层描述、标准孔地层描述、遗迹结论 | 汇总主要地层和遗迹情况生成 | 人工填写可覆盖。 |
| `{{文物概况待补说明}}` | 文物概况派生 | 回函标题、文号或结论不完整时生成 | 用于避免空白回函句。 |

回函识别维护规则：文件名中含 `文旅/文化旅游/文化和旅游/文物/复函/函`、且含“关于”或文号特征的 PDF，即使因 `坐标/范围` 等词导致文件名初筛分数较低，也要读取正文或 OCR 文本后重新判断。扫描件回函优先使用本地 PaddleOCR，PaddleOCR 不可用时回退 RapidOCR、tesseract 等后备 OCR。若项目资料中能识别到文物部门回函，但填报表 `文物概况类型` 仍为 `无回函且未发现文物` 或 `无回函但涉及文物`，专项检查应归入 `回函信息待复核`。

## 维护说明

1. 新增模板变量时，应同步更新本文档，并确认变量来源属于人工填写表、制图成果表格、外业记录、图片槽位或派生变量之一。
2. 新增图片槽位时，应同步更新 `智能生成报告技能资料/工具脚本/fill_smart_template_from_form.py` 的 `build_image_slots()` 匹配规则，并在本文档“图片与图件变量”中补充路径/关键词规则。
3. 基准模板视为只读，不能为了删除某张图或调整某个项目输出而修改基准模板。项目级图片删除、题注删除和图号重排应发生在生成后的 DOCX 中。
4. 本文档属于长期项目说明，保存在 `智能生成报告技能资料/项目说明/`。过程诊断、临时检查和生成检查 txt/md 应放入 `过程资料/`，不得留在 `生成报告/<项目名称>/`。
5. 如果通过 DOCX XML 扫描占位符时出现带 XML 片段的畸形 `{{图:...}}`，通常是 Word 将同一个占位符拆成多个 run 造成的。维护时以模板中肉眼可读的规范变量名和生成脚本支持的变量名为准，不逐字记录畸形 XML 文本。
